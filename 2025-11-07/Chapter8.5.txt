CHAPTER 8:  EXCEPTIONAL CONTROL FLOW

    V. Signals

        • A signal is a small message that notifies a process that an event of some type has occured in the system

            - for example, if you type "CTRL+C" while a process is running in the foreground, then the kernel sends a SIGINT to each process
              in the foreground process group

              when a child process terminates or stops, the kernel sends a SIGCHILD signal to the parent

        A. Signal Terminology

            • The transfer of a signal to a destination process occurs in two distinct steps:

                (1) sending a signal: the kernel delivers a signal to a destination process by updating some state in the context of the destination process

                (2) receiving a signal: a destination process receives a signal when it is forced by the kernel to react in some way to the delivery of the signal.

                    - the process can either ignore the signal, terminate, or catch the signal by executing a user-level function called a signal-handler

            • A signal that has been sent but not yet received is called a 'pending signal'

            • A process can selectively block the receipt of certain signals

                - a blocked signal may be delivered but the resulting pending signal is not received by destination process untilt he process unblocks the signal

            • For each process, teh kernel maintains the set of pending signals in the pending bit vector, and the set of blocked signals in the blocked bit 
              vector (aka signal mask)

                - the kernel sets bit k in pending whenever a signal of type k is delivered and clears bit k in pending whenever a signal of type
                  k is received

        B. Sending Signals

            • Unix systems provide a number of mechanisms for sending signals to processes, and all mechanisms rely on the notion of a 'process group'.

            • Every process belongs to exactly one process group, and the getpgrp() funciton returns the process group ID of the current process

                pid_t getpgrp(void)

                - by default, a child belongs to the same process group of its parent
            
            • A process can change the process group of itself or another process by using the setpgid() function

                int setpdgid(pid_t pid, pid_t pgid);

                - the easiest way to understand this is to know that 0 is a special shortcut for the kernel:

                    -- when 0 is used for pid (the process you want to change) it means the current calling process

                    -- when 0 is used for pgid (the target group ID) it means to create a new group and use the pid of the process you're changing as the new PGID

            • The /bin/kill program sends an arbitrary signal to another process

                - example: /bin/kill -9 15213 sends signal 9 (SIGKILL) to process 15213

                - using a negative pID causes the signal to be sent ot every process in the process group

                - example: /bin/kill -9 -15213 sends a SIGKILL signal to every process in the process group 15213

            • The shell uses the term 'job' as a convenient way to refer to the entire collection of processes that originated from a single command line entry.

                - The shell creates a separate process group for each job

                - example: When you type 'ls | sort', you are telling the shell to create one job:

                    -- this job contains two processes (one running ls and one running sort)

                    -- these two processes are put into a single process group so they can be managed (like being stopped or killed) as one unit

                - see figure 8.28 on page 786 for a visual of this

            • Processes send signals to other processes by calling the kill() function

                int kill(pid_t pid, int sig)

                - if pid > 0, that pid is targeted.  

                - if pid = 0, then kill() sends a signal 'sig' to every process in the process group of the calling process (including the calling process itself)

                - if pid < 0, the kill() sends signal 'sig' to every process in the process group |pid|

            • A process can send SIGALARM signals to itself by calling the alarm function:

                unsigned int alarm(unsigned int secs)

                - this function arranges for the kernel to send a SIGALARM signal to the calling process in 'secs' seconds.

                - returns the number of seconds remaining until any pending alarm was due to be delivered

        C. Recieving Signals

            • When the kernel switches a process 'p' from kernel mode to user mode, it checks the set of unblocked pending signasl (pending & ~blocked) 
              for p

                - if the set is empty, the kernel gives control back to the process at the next instruction

                - if the set is non-empty, the kernel chooses one signal (typically the one with the lowest signal number) and forces the process to receive it

                    -- if the process has defined a custom handler for that signal, the kernel forces the process to run that handler,
                       else the default disposition for that signal is carried out

            • A process can modify the default action associated with a signal by using the signal() function

                1. typedef void (*sighandler_t)(int);
                2. sighandler_t signal(int signum, sighandler_t handler);

                - line 1 defines a new type called 'sighandler_t': this new type is an alias for "a pointer to a function that takes an int and returns void"

                    -- the set of parentheses immediately following the pointer name (int) is the crucial part. In C, if you have a variable declaration followed by 
                       a set of parentheses containing arguments, the compiler knows this variable must be a pointer to a function: it specifies the function's parameter list

                    -- the type preceding the whole expression specifies what the function must return: (void)

                    The compiler looks at the entire structure:

                    -- "I am creating an alias (typedef) called sighandler_t. Any variable declared as sighandler_t must be a pointer (*) to a function() that takes one 
                        integer argument (int) and returns nothing (void)

                - in line 2, the signal() function returns a pointer to the old handler function (the address of the function that was previously handling signal 'signum' or of 
                  the Default Action (SIG_DFL) or Ignore Action (SIG_IGN))

                    -- if 'handler' is SIG_IGN, then signals of type signum are ignored

                    -- if 'handler' is SIG_DFL, then the action for signals of type signum reverts to the default action

                    -- otherwise, handler is the address of a user-defined function, called a 'signal handler', that will be called whenever the process receives a signal
                       of type 'signum'
                       
                    -- changing the default action by passing the address of a handler to the signal function is known as 'installing the handler'

                    -- the invocation of the handler is called 'catching the signal'
                    
                    -- the execution of the handler is referred to as 'handling the signal'

            • As we discussed, the signal() function usually returns the address of the old handler. However, if the function fails for any reason 
              (e.g., trying to set a handler for the unchangeable signals SIGKILL or SIGSTOP), it has a specific way of telling the program via an error signal.

                - if an error occurs, the function does not return the old handler's address: instead, it returns a special constant value defined in 
                  the C library: SIG_ERR

                - this information is important for understanding the code on page 789

        D. Blocking and Unblocking Signals        

            • Linux provides implicit and explicit mechanisms for blocking signals:

                (1) implicit blocking mechanism: by default, the kernel blocks any pending signals of the type currently being processed by a handler

                (2) explicit blocking mechanism: applications can explicitely block and unblock selected signals using the 'sigprocmask' function and its helpers

                int sigprocmask(int how,const sigset_t *set, sigset_t *oldset);

                - a sigset_t variable (set in this case) is essentially a bit vector where each bit corresponds to a specific signal number (like SIGINT, SIGHUP, etc.)  
                
                - if a bit is set (1), the signal is included in the set

                - you use helper functions to manipulate these sets:
                
                    -- sigemptyset(set) - initializes the set to be empty (all bits are 0)
                    -- sigfillset(set) - initializes the set to include every possible signal (all bits are 1)
                    -- sigaddset(set, signum) - adds a single signal (signum) to the set (sets the corresponding bit to 1)
                    -- sigdelset(set, signum) - removes a single signal (signum) from the set (sets the corresponding bit to 0)
                    -- sigismember(set, signum) - checks if a signal is currently in the set

                - The sigprocmask() function uses the signal set you create to change the process's internal blocked bit vector:

                    (1) the 'how' argument specifies how the blocking set should be modified:

                        -- SIG_BLOCK: add the signals in the set to the currently blocked signals
                        -- SIG_UNBLOCK: remove the signals in the set from the currrently blocked signals
                        -- SIG_SETMASK: replace the currently blocked signals entirely with the signals in the set

                    (2) the 'set' argument is a pointer to the sigset_t structure containing the signals to add, remove or set

                    (3) the 'oldset' argument is a pointer to a sigset_t structure where the previous set of blocked signals will be saved:
                        if NULL, the previous set is discarded

            • Below we can visualize what is happening with sigset_t set as a bit vector:

                +---------------+--------+--------+--------+--------+--------+
                | Signal Number |   1    |   2    |   3    |   4    |   5    |
                +---------------+--------+--------+--------+--------+--------+
                | Signal Name   | SIGHUP | SIGINT | SIGQUIT| SIGILL | SIGTRAP|
                +---------------+--------+--------+--------+--------+--------+
                | Mask          |   0    |   1    |   0    |   0    |   0    |
                +---------------+--------+--------+--------+--------+--------+
                | Blocked       |   0    |   0    |   0    |   1    |   0    |
                +---------------+--------+--------+--------+--------+--------+

                
                (1) currently the sigset_t mask has the bitvector 01000 and blocked has 00010, meaning SIGINT is in our 'mask' set and SIGILL is currently blocked

                (2) we can add members to our set via one the helper functions:

                    - sigaddset(&mask, SIGQUIT)

                    - now sigset_t mask has the bitvector 01100

                (3) when we call sigprocmask(SIG_BLOCK, &mask, &prev_mask), we add the mask to currently blocked signals
                    via (blocked | mask)

                (4) now SIGINT and SIGILL are blocked, so 'blocked' is now 011010

                (5) consider we also had a sigset_t prev_mask bitvector as well as 'mask', and then we ran the following code:
                  
                    sigprocmask(SIG_BLOCK, &mask, &prev_mask)
                    
                    now the previous old blocked mask (00010) is saved in prev_mask, so when we run 

                    sigprocmask(SIG_SETMASK, &prev_mask, NULL), we restore the old blocked bitvector to its original state (00010)
                
        E. Writing Signal Handlers

            • Signal handlers run “concurrently” because:

                (1) they interrupt your code at unpredictable times

                (2) they run asynchronously

                (3) they share the same memory as the main program

                (4) they don’t follow your program’s normal control flow

            • Here are some general guidelines for writing safe signal handlers:

                (1) keep them as simple as possible (ex: set a global flag then return immediately)

                (2) call only async-signal-safe functions in your handlers 
                
                    - these are functions that only access local variables (reentrant) or cannot be interuppted by a signal handler

                    - the only safe way to generate output from a signal handler is using the write() function as printf and sprintf are unsafe

                * see figure 8.33 on page 794 for a list of async-signal-safe functions

                (3) save and restore errno inside the handler

                    - this can look something like this: 

                    void handler(int sig) 
                    {
                        int old_errno = errno;   // save errno used by main() function

                        write(1, "Signal!\n", 8); // possibly generates a new errno if write() fails

                        errno = old_errno;       // restore the old errno for the main() function
                    }

                    - this is not necessary if the handler terminates by calling exit()

                (4) protect access to shared global data structures by blocking all signals

                    - consider this example below:

                    int counter = 0;    // global non-atomic variable (interruptable)
                    /* somewhere in main */
                    counter++;

                    conceptually, the CPU is executing assembly commands:

                    1. mov    counter(%rip), %rax     ; load the value at the address (RIP + offset_to_counter) into %rax
                    2. add    $1, %rax                ; add 1 to the register %rax
                    3. mov    %rax, counter(%rip)     ; store register value back into memory

                    lets assume a signal were to arrive after line 2:

                    1. mov    counter(%rip), %rax     
                    2. add    $1, %rax     

                    ---- SIGNAL ARRIVES HERE ----

                    SIGNAL HANDLER:
                        counter = 0;             
                        return;

                    ---- SIGNAL HANDLER RETURNS ----

                    3. mov %rax, counter(%rip) // writes the old value (1) back to memory

                    this is a problem because the handler never reset the value of the counter, the main function only ever saw the 
                    stale value of 'counter'

                    THIS IS KNOWN AS A RACE CONDITION because both main() and the signal handler are "racing" to write to memory (whichever one writes LAST, wins
                    even if that produces an incorrect value)

                    - we can block signals to prevent race condtions:

                        sigprocmask(SIG_BLOCK, &all_signals, &old);

                        // any code here cannot be interrupted 
                        counter++;

                        // unblock the signals
                        sigprocmask(SIG_SETMASK, &old, NULL);

                (5) declare flags with sig_atomic_t

                    C provides an integer datatype, sig_atomic_t, for which reads and writes are guaranteed to be atomic (uninterruptible) because 
                    they can be implemented with a single instruction

                        - since they cant be interrupted, you can safely read from and write to sig_atomic_t variables without having to temporarily 
                          block signals                       
                        
            • One of the nonintuitive aspects of signals is that pending signals are not queued.

                - because the pending bit vector contains exactly one bit for each type of signal, there can be at most one pending signal of any particular type

            • If two signals of type k are sent to a destination process while signal k is blocked because the destination process is currently executing a 
              handler for signal k, then the second signal is simply discarded; it is not queued.

                - the key idea is that the existence of a pending signal merely indicates that at least one signal has arrived

            • Reference the buggy code on page 797 to understand the discussion below:

                (1) inside handler1, the function reaps children using waitpid(-1, NULL, 0), which means "wait for any child process to change state; 
                    block until one does"

                    the handler prints a message, sleeps for 1 second (exascerbating the race condition) and restores errno

                (2) the signal handler is installed with the signal() function via signal(SIGCHLD, handler1)

                    if this function call returns SIG_ERR, then the call to signal failed and the function unixerror() is called

                    * recall unixerror() has the following implementation:

                    void unix_error(char *msg)
                    {
                        fprintf(stderr, "%s: %s\n", msg, strerror(errno));
                        exit(0);
                    }

                (3) the parent creates the children with Fork() wrapper function

                (4) the line read(STDIN_FILENO, buf, sizeof(buf)) reads from fd = 0 (stdin) into a buffer called 'buf', for which we need the size

                    the result of the read is stored in a variable 'n', and if n < 0, then a read error has occured, so we call the 
                    unixerror() function again

                (5) the parent simply processes input with an infinite while loop until the user interrupts with CTRL+c

            • The problem with the above program is that only 2 of the three children are reaped.

                - the code failed to account for the fact that signals are not queued

                - the first signal is caught and handled, the second signal becomes pending and will be handled after the first handler finishes, but the 
                  third signal arriving while the handler is still running is discarded

            • To fix this, we can simply add a while loop with the condition waitpid(-1, NULL, 0) > 0 is true, reap the children, then check if
              errno != ECHILD

                - waitpid returns -1 when there are no more children to reap, ECHILD is the expected error in that case
                
                - any other error is reported

                - the handler doesn’t rely on receiving multiple signals; it just reaps all terminated children each time it runs because 
                  waitpid(-1, NULL, 0) can reap any child that has terminated, regardless of whether the SIGCHLD signal was lost or not

            • Some older Unix systems restore the action for signal 'k' to its default after signal 'k' has been caught by a handler. 

                - on these systems, the handler must explicitly reinstall itself, by calling signal, each time it runs

            • On some older versions of Unix, slow system calls that are interrupted when a handler catches a signal do not resume when the signal handler returns, 
              but instead return immediately to the user with an error condition and errno set to EINTR.

            • To deal with these issues, the POSIX standard defines the sigaction() function, which allows users to clearly specify the signal-handling semantics they want 
              when they install a handler.

              int sigaction(int signum, struct sigaction *act, struct sigaction *oldact)

            • the sigaction() function is unwieldly because it requires the user to set the entries of a complicated structure, so we can create a wrapper 
              function called Signal() that calls sigaction() for us

                handler_t *Signal(int signum, handler_t *handler)
                {
                    // declare the struct sigaction variables
                    struct sigaction action, old_action;

                    // assigns the function pointer passed to the wrapper to the handler field
                    action.sa_handler = handler;
                    
                    // initializes the mask to empty so no extra signals are blocked while the handler runs (except the signal itself)
                    sigemptyset(&action.sa_mask);  
                    
                    // set flags: SA_RESTART ensures interrupted slow syscalls restart automatically
                    action.sa_flags = SA_RESTART; 
                    
                    // install the new handler and save the old handler
                    if (sigaction(signum, &action, &old_action) < 0) 
                    {
                        unix_error("Signal error");
                    }
                    
                    // return the address of the old handler (for reversibility)
                    return (old_action.sa_handler);
                }
        
        F. Synchronizing Flows to Avoid Nasty Concurrency Bugs

            • Race conditions happen because we can’t predict all the ways instructions from concurrent flows interleave, so we need synchronization to 
              guarantee correctness.

            • At first glance, the code on page 803 looks like fine, but upon deeper inspection we notice that it has a subtle flaw:

                (1) the parent executes the fork() function and the kernel schedules the newly created child to run instead of the parent
                
                (2) before the parent is able to run again, the child terminates and becomes a zombie, causing the kernel to deliver a SIGCHLD signal to the parent
                
                (3) later, when the parent becomes runnable again but before it is executed, the kernel notices the pending SIGCHLD and causes it to be received by 
                    running the signal handler in the parent
                    
                (4) the signal handler reaps the terminated child and calls deletejob(), which does nothing because the parent has not added the child to the list yet
                    
                (5) after the handler completes, the kernel then runs the parent, which returns from fork and incorrectly adds the (nonexistent) child to the job list by 
                    calling addjob()

                * crucially, adding and deleting jobs happen atomically and safely without interference from signals that might arrive at the same time, preventing 
                  obvious race conditions, but more subtle race condtions are still present

            • There are also interleavings where events occur in the correct order:
            
                - if the kernel happens to schedule the parent to run when the fork call returns instead of the child, then the parent will correctly add
                  the child to the job list before the child terminates and the signal handler removes the job from the list

            • Thus, for some interleavings of the parent’s main routine and signal-handling flows, it is possible for deletejob() to be called before addjob().

                - for this race condtion, if addjob() wins the race, then the answer is correct, but if deletejob() wins, the answer is incorrect

            • Such errors are enormously difficult to debug because it is often impossible to test every interleaving.

            • Figure 8.40 on page 805 shows one way to eliminate this race condition, copied below:

                void handler(int sig)
                {
                    int olderrno = errno;
                    sigset_t mask_all, prev_all;
                    pid_t pid;

                    Sigfillset(&mask_all);

                    while ((pid = waitpid(-1, NULL, 0)) > 0) { 
                        Sigprocmask(SIG_BLOCK, &mask_all, &prev_all);
                        deletejob(pid);                     
                        Sigprocmask(SIG_SETMASK, &prev_all, NULL);
                    }

                    if (errno != ECHILD)
                        Sio_error("waitpid error");

                    errno = olderrno;
                }

                int main(int argc, char **argv)
                {
                    int pid;
                    sigset_t mask_all, mask_one, prev_one;

                    Sigfillset(&mask_all);
                    Sigemptyset(&mask_one);
                    Sigaddset(&mask_one, SIGCHLD);

                    Signal(SIGCHLD, handler);
                    initjobs();                           

                    while (1) {

                        Sigprocmask(SIG_BLOCK, &mask_one, &prev_one);   /* Block SIGCHLD before the call to fork() */

                        if ((pid = Fork()) == 0) {          
                            Sigprocmask(SIG_SETMASK, &prev_one, NULL);  /* Unblock SIGCHLD only after the call to fork()*/
                            Execve("/bin/date", argv, NULL);
                        }

                        Sigprocmask(SIG_BLOCK, &mask_all, NULL);        
                        addjob(pid);                        
                        Sigprocmask(SIG_SETMASK, &prev_one, NULL);      
                    }

                    exit(0);
                }

                - nothing changes in the handler, but in main we add an additional set of blocking and unblocking:

                    (1) block all signals before the call to fork(): this ensures the parent cannot receive a SIGCHLD between the fork() and the addjob() call,
                        preventing deletejob() from being called before addjob()

                    (2) only unblock the signals after the parent has created the child: this ensures SIGCHLD is only delivered after the parent has 
                        added the job to the list, so the handler sees an entry it can safely delete

                    (3) also, remember that that children inherit the blocked set of their parents, so we must be careful to unblock the SIGCHLD signal in the child 
                        before calling execve()
                        
                        if SIGCHLD is blocked before the child becomes a new parent after execve(), any children that new program creates will 
                        send SIGCHLD to their parent (the exec’d program), but since SIGCHLD is blocked in that parent, it will never handle those signals

                        this means its own children become zombies forever because that that program never reaps them

        G. Explicitely Waiting for Signals

            • Sometimes a main program needs to explicitly wait for a certain signal handler to run. 
            
                - for example, when a Linux shell creates a foreground job, it must wait for the job to terminate and be reaped by the SIGCHLD handler 
                  before accepting the next user command

            • Reference the code in figure 8.41 on page 806 to understand the discussion below:

                (1) the parent installs handlers for SIGINT and SIGCHLD and then enters an infinite loop. 
                
                (2) it blocks SIGCHLD to avoid the race between the parent and child that we observed in figure 8.39

                (3) after creating the child, it resets the global flag 'volatile sig_atomic_t pid' to zero, unblocks SIGCHLD, and then waits in a 
                    spin loop for 'pid'to become non-zero (waits for the signal handler to set the flag (aka reap the children))

                (4) this terminates the spin loop, and the parent continues with additional work before starting the next iteration

            • While this codeis correct, the spin loop is wasteful because the CPU burns cycles doing:

                pid? pid? pid? pid? pid? pid?

            • You might be tempted to think that instead of looping and wasting CPU, we can put the process to sleep via the pause() funciton until 
              something happens

                while (!pid)
                pause();

                - but this actually introduces a race condition where a SIGCHLD signal can be received after the while-loop test but before the pause, causing
                  the program to hang indefinitely

            • Another option is to replace the pause with sleep:

                while (!pid) 
                sleep(1);

                - the problem with this approach is that it is simply too slow: 1 second is an ETERNITY relative to the CPU executing BILLIONS of instructions
                  per second

                - Why not just pick a “good enough” number?

                    -- too small means high CPU usage (wasteful)

                    -- too large means slow response (lags)

                    -- There is no “perfect” number that is both fast and efficient

            • The proper solution is to use sigsuspend():

                int sigsuspend(const sigset_t *mask);

            • sigsuspend() does three things:
                
                (1) temporarily replaces the current blocked set with the mask you provide (unblocks SIGCHLD in this case)

                (2) suspends the process until a signal arrives whose action is either to run a handler or terminate the process

                    - if the action is to terminate, then the process terminates without returning from sigsuspend()

                (3) after the handler runs, sigsuspend() returns AFTER THE HANDLER RETURNS (thus achieving the goal of waiting for the signal handler) and 
                    restores the old blocked set automatically

                * see figure 8.42 on page 808 to see the final solution that utilizes sigsuspend(&prev)

            • The sigsuspend() function is equivalent to an atomic (uninterruptible) version of the following:

                sigprocmask(SIG_BLOCK,&mask,&prev);
                pause();
                sigprocmask(SIG_SETMASK,&prev,NULL);

                - since the above code requires three instructions, it is NOT atomic and thus interruptable (i.e. vulnerable to race conditions)

                - sigsupsend() does the above 3 instructions in one single atomic instruction, so it is uninterruptable and thus completely avoids
                  race conditions
