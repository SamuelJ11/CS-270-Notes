        • Formally, address translation is a mapping between the elements of an N element virtual address space (VAS) and an M-element physical address space (PAS):

            - MAP:VAS → PAS ∪ ∅

            - if data at virtual address A is present at physical address A', else it maps to ∅

        • Reference figure 9.12 on page 840 to visualize the discussion below:

            - a control register in the CPU called the 'page table base register' (PTBR) points to the current page table

            - the CPU uses the address in the PTBR to find the physical memory address where the corresponding PTE for a VPN is stored:

                PTRB + (VPN x sizeof(PTE)) 

            - the n-bit (usually 32 or 64) virutal address has two components:

                -- p-bit virtual page offset 
                -- (n - p)-bit virtual page number

            - the MMU uses the VPN AS AN INDEX INTO THE PAGE TABLE to select the appropriate PTE:

                -- for example, VPN 0 selects PTE 0, VPN 1 selects PTE 1, etc.

            - the corresponding physical address is the concatenation of the physcial page number from the PTE and the VPO from 
              the virtual address

                -- since the size of a page is the same in both the virtual and physical address spaces, the PPO is the same as the VPO

        A. Integrating Caches and VM

            • Most systems use physical addresses to access the SRAM cache as access rights are checked as part of the address 
              translation process

            • Reference figure 9.14 on page 842 to understand the following below:

                - the Processor generates a Virtual Address (VA), which is the address the program thinks it is using

                - the VA is sent to the MMU (Memory Management Unit), which is hardware located on the CPU chip

                - the MMU must calculate the address of the specific Page Table Entry (PTE) it needs. This address is the PTEA (Page Table Entry Address)
                  calculated as before via PTRB + (VPN x sizeof(PTE))

                - the PTEA (which is a physical address pointing to the PTE in DRAM) is sent out

                    - PTE's are cached in the L1 cache

                - if the PTE is found in the cache, the PTE is quickly returned to the MMU

                    -- if the PTE is not found in the cache, a request is sent to memory to fetch the PTE.  the PTE is then returned
                       to the MMU and simultaneously cached in the L1 cache for future use

                - once the MMU recieves the PTE, it checks the valid bit in the PTE struct, and if the page is valid and in memory, the MMU extracts the
                  Physical Page Number (PPN) from the PTE struct.

                - then the MMU combines the PPN with the VPO to construct the final PA

                - finally, the PA is sent to the L1 cache with which the data will be looked up:

                    -- if the data is found in the L1 cache, it is immediately sent back to the processor

                    -- if the data is not in the L1 cache, the PA is forwarded to memory which retrieves the requested data,
                       which is then sent back to the L1 cache and simultaneously forwarded to the CPU

        B. Speeding Up Address Translation with a TLB

            • If the PTE happens to be cached in L1, then the cost goes down to a handful of clock cycles, but many modern systems to eliminate
              even this cost by including a small cache of PTE's in the MMU called a 'translation lookaside buffer' (TLB).

            • A TLB is a small, virtually addressed cache where each line holds a block consisting of a single PTE

                                    Virtual Address (VA)

                n-1                  p+t p+t-1                 p p-1                      0
                +-----------------------+-----------------------+-------------------------+
                |     TLB tag (TLBT)    |    TLB index (TLBI)   |       VPO               |
                +-----------------------+-----------------------+-------------------------+
                \______________________________________________/
                                VPN

            • As shown in the figure above, the index and tag fields are extracted from the virtual page number and are used for set 
              selection and line matching.           

            • If the TLB has T = 2ᵗ sets, then the TLB index (TLBI) is the t least-significant bits of the virtual page number (VPN), and the TLB tag (TLBT) is 
              all the remaining bits of the VPN.

            • Reference figure 9.16(a) to see the process of a cache hit for the TLB:

                (1) The CPU generates a virtual address

                (2) the MMU fetches the appropriate PTE from the TLB

                (3) the MMU translates the virtual address to a physical address via extracting the PPN from the PTE struct and concating this with this VPO, and this
                    PA is then sent to the cache/main memory

                (4) the cache/main memory returns the requested data word to the CPU

            • When there is a TLB miss, the MMU must fetch the PTE from teh L1 cache.

        C. Multi-Level Page Tables

            • A single-level page table creates a memory allocation problem because it must contain one entry for every single possible virtual page number.

            • In a standard 32-bit system with 4 KB pages, the system is capable of indexing roughly 1 million distinct pages (specifically, 2^20 pages).

            • Because a single-level table is just one giant array, the Operating System is forced to allocate memory for all 1 million entries right at the start.

            • This results in 4 MB of contiguous page table memory being consumed for every single process running on the computer.

            • Let’s look at the specific math for a 32-bit system to see where that "4 MB" number comes from:

                - the processor uses 32 bits for an address

                - the system chops memory into 4 KB pages. Since 4 KB is equal to 2^12 bytes, the bottom 12 bits of the address are used for the offset

                - this leaves the remaining bits for the Page Number: 32 total bits - 12 offset bits = 20 bits

                - therefore, there are 2^20 (approximately 1 million) unique page numbers that need to be mapped

                - calculating the total size in bytes:

                    -- each entry (PTE) in the table is typically 4 bytes in size (32 bits wide)
                    -- we have 2^20 entries (1,048,576 entries)
                    -- the math is: 4 bytes x 1,048,576 entries = 4,194,304 bytes
                    -- 4,194,304 bytes = exactly 4 MB (the size of the Single-Level Page Table)

            • However, in a two-level system, we change the meaning of the first entry. A Level 1 PTE does NOT point to the actual data (the physical page frame); 
              instead, it points to the start of a Level 2 Page Table.

            • Think of the Level 1 table as a "Directory" or a "Table of Contents.":
            
                - it doesn't tell you where the data is; it tells you which "Chapter" (Level 2 table) to look in.

            • The fundamental design goal is for the Level 2 page table to fit exactly inside one standard page of memory.

                - this is crucial for memory management; if a page table is the exact same size as a standard page (4 KB), the Operating System can easily 
                  swap the page table itself in and out of memory, just like it swaps regular data

            • We thus compute the number of PTEs that fit into a Level 2 table:

                - we have a standard page size of 4 KB (4096 bytes) available to hold the table

                - each individual entry (PTE) is 4 bytes

                - the math is: 4096 bytes / 4 bytes = 1024 entries

                - therefore, one Level 2 Page Table contains exactly 1,024 Page Table Entries

                - each of these 1,024 entries maps a standard 4 KB virtual page of actual user data

                - once you are at Level 2, the PTE works just like a normal one: it points to the physical RAM where your data lives

            • Because one Level 1 PTE points to exactly one Level 2 table, that single Level 1 PTE is effectively "responsible" for mapping a massive chunk of memory

            • Let’s calculate the coverage of that single Level 1 entry:

                - it points to 1 Level 2 table
                - that Level 2 table contains 1,024 entries
                - each of those entries points to 4 KB of data
                - the math is: 1,024 entries x 4 KB per entry = 4,096 KB
                - 4,096 KB = exactly 4 MB. So, one entry in the Level 1 table covers a 4 MB "chunk" of the virtual address space

            • Finally, we determine how many Level 1 entries (and potential Level 2 tables) we need to cover the whole system:

                - the total virtual address space is 4 GB (4,096 MB)
                - each Level 1 entry covers a 4 MB chunk
                - the math is: 4,096 MB total space / 4 MB per chunk = 1,024 chunks
                - this means there are 1,024 Level 1 PTEs (not 4,096)
                - consequently, there are 1,024 potential Level 2 page tables, though we only create the ones we actually need

            • This scheme reduces memory requirements in two ways:
            
                - first, if a PTE in the level 1 table is null, then the corresponding level 2 page table does not even have
                  to exist, representing a significant potential savings, since most of the 4 GB virtual address space for a typical program is unallocated

                - Second, only the level 1 table needs to be in main memory at all times

            • We can generalize a 2-level heirachy to a k-level heirachy:

                - in a k-level system, the Virtual Address (VA) is chopped up into more pieces (instead of just one VPN and one VPO, the VPN is split into k 
                  different parts)
                
                - The PTE in the final level-k table contains the actual Physical Page Number (PPN) (or a disk address if the page is swapped out)
        
        D. Putting It Together: End-to-End Address Translation

            • For this example, we make the following assumptions:

                - The memory is byte addressable
                - Memory accesses are to 1-byte words (not 4-byte words)
                - Virtual addresses are 14 bits wide (n = 14)
                - Physical addresses are 12 bits wide (m = 12)
                - The page size is 64 bytes (P = 64)
                - The TLB is 4-way set associative with 16 total entries
                - The L1d-cache is physically addressed and direct mapped, with a 4-byte line size and 16 total sets

            • Quick check: the virtual address space is 2^14 = 16384 bytes, and each page is 2^6 bytes, so a hyptothetical page table would have 2^8 PTE's.

            • Since each page is 2⁶ = 64 bytes, the low-order 6 bits of the virtual and physical addresses serve as the VPO and PPO, respectively. 
            
            • The high-order 8 bits of the virtual address serve as the VPN, and the high-order 6 bits of the physical address serve as the PPN.

            • To understand the figures on page 847, it is helpful to first do a walkthrough with an example virtual address:

                - consider the CPU executes a load instruction that reads the byte at address 0x03d4

                (1) Break Down the Virtual Address

                    -- convert the 14-bit virtual address to binary: 00 0011 1101 0100

                    -- split this binary string into the VPN and VPO: 

                       VPO (lower 6 bits): 010100   (0x14)
                       VPN (upper 8 bits): 00001111 (0x0F)

                    -- current status: we want page 0x0F at offset 0x14, but we need to find the PPN for VPN 0x0F

                (2) The TLB lookup

                    - here the MMU first checks the TLB to see if we have cached this page (0x0F) recently

                    - to look into the TLB, we have to split the VPN further based on the rules from the text

                        -- TLB Index is the lower 2 bits of the VPN: 11 (TLBI = 3)

                        -- TLB Tag is the remaining upper 6 bits of the VPN: 00 0011 (TLBT = 0x03)

                    - now look at figure 9.20(a):

                        (1) go to set 3
                        (2) scan the tags in that row for tag 0x03
                        (3) check the valid bit, since its 1 we have a hit
                        (4) read the PPN value in this entry: PPN = 0x0D

                    - current status: we now know that the virtual page 0x0F lives at PP 0x0D, and crucially, we didn't have to reference the 
                                      relatively slow page table in memory (figure 9.20b)

                (3) Construct the Physical Address

                    - the MMU now constructs the PPN by concatenating the PA with the original VPO:

                        -- PPN: 0x0D -> 001101
                        -- VPO: 0x14 -> 010100

                        PPN || VPO = 001101010100 (0x354)

                (4) The Cache Lookup (figure 9.20c)

                    - Now we need to actually fetch the data byte from the L1 Cache using the PA calculated earlier

                    - Since the cache is physically addressed, we split these 12 bits:

                        -- Block Offset (CO): bottom 2 bits -> 00 (0x0)       
                        -- Cache Index (CI): the next 4 bits -> 0101 (0x5)
                        -- Cache Tag (CT): the upper 6 bits -> 001101 (0x0D)

                    - Now go to set index 5, look at the tag corresponding to 0x0D, and check the valid bit

                        - valid bit = 1, so this is a cache hit

                    - Use the block offset to find the specific byte

                        - column 'Blk 0' contains the value 0x36

                (5) Final result

                    - the byte stored at virtual address 0x03D4 is 0x36
