CHAPTER 10:  System-Level I/O

    • An input operation copies data from an I/O device to main memory, and an output operation copies data from memory to a device

    • All languages have “higher-level” I/O
    
        - every programming language gives you convenient functions for input/output

        - in C, you have printf, scanf, fopen, etc

        - in C++, you have cout << and cin >>

    • These are easy to use because they are buffered and abstract away details (under the hood, these functions still rely on Unix’s low-level I/O)

        - on Linux, those high-level functions are not magic—they are built on top of low-level system calls provided by the kernel:

        - read()
        - write()
        - open()
        - close()

        - so even if you use cout << "Hello"

        - eventually the C++ library calls write() to actually send characters to the terminal

    •  Sometimes you have no choice but to use Unix I/O:

        - for example, the standard I/O library provides no way to access file metadata such as file size or file creation time. Further, there are problems
          with the standard I/O library that make it risky to use for network programming

    A. Unix I/O

        • A linux file is a sequence of 'm' bytes:

            - B₀, B₁, . . . Bₖ, Bₘ₋₁ 

        • In Linux (and all Unix-like systems), devices are not special objects.

            - a disk is represented as a file

            - a keyboard is represented as a file

            - a network connection is represented as a file

            - even things that aren’t really files, like /dev/null or pipes, are treated as files

        • This means if you can read/write to a file, you can read/write to almost any device.

            - when a program wants to use a file or device (like a disk, terminal, or network connection), it must first open it using the 
              system-level open() function

                - open() returns a non-negative integer called a 'file descriptor' (fd)

            - after opening the file, all operations use just the descriptor (the kernel looks up all the details of that file using the descriptor)

        • Each process created by a Linux shell begins life with 3 open files:

            (1) standard input (fd = 0)
            (2) standard output (fd = 1)
            (3) standard error (fd = 2) 

            - the header file <unistd.h> defines constants STDIN_FILENO, STDOUT_FILENO, and STDERR_FILENO for your convienence so you don't have to use 
              explicit descriptor values

        • The kernel maintains a file position 'k' (initially 0) for each open file.

            - an application can set the current file position 'k' explicitly by performing a 'seek' operation

        • A 'read' operation 'n > 0' bytes from a file to memory, starting at the current file position 'k' and then incrementing 'k' by n.

            - given a file with 'm' bytes, a read operation when k >= m triggers a condition known as end-of-file (EOF)

        • A 'write' operation copies n > 0 bytes from memory to a file, starting at the current file position 'k' and then updating 'k'

        • When an application informs the kernel that a file is being closed, the kernel responds by freeing the data structures it created when the 
          file was opened and restores the fd to a pool of available file descriptors

    B. Files 

        • Each linux file has a 'type' that indicates its role in the system:

            - regular files contain arbitrary data

                -- text files contain only ASCII or Unicode characters, where each line is a sequence of characters terminated by a 'newline' character
                   ('\n') and has the numeric value 0x0a

                -- binary files are everything else

            - directories are files consisting of an array of 'links', where each link maps a filename to a file (which may be another directory)

                -- each directory contains at least two entries: 

                    (1) . is a link to the directory itself
                    (2) .. is a link to the parent directory

            - a socket is a file that is used to communicate with another process across a network

        • The Linux kernel organizes all files in a single directory hierarchy anchored by the root directory name '/'

        • Each process has a current working directory that identifies its current location in the directory heirarchy shown on page 915 

        • Locations in the directory hierarchy are specified by absolute or relative pathnames

    c. Opening and Closing Files

        • A process opens an existing file or creates a new file by calling the open function

        • The open function converts a filename to a file descriptor and returns the descriptor number.

            - the flags argument indicates how the process intends to access the file:

                (1) O_RDONLY is read only
                (2) O_WRONLY is write only
                (3) O_RDWR is read and write

            - the flags argument can be ORed with one or more bit masks that provide additional instructions for writing:

                (1) O_CREATE - if the file doesn't exist, create a truncated (empty) version of it
                (2) O_TRUNC - if the file already exists, truncate it
                (3) O_APPEND - before each write operation, set the file position to the end of the file

            * see page 917 for access permission bits

        • The 'mode' argument specifies the access permission bits of new files

        • suppose we are given the following default values for mode and umask:

            #define DEF_MODE    S_IRUSR|S_IWUSR|S_IRGRP|S_IWGRP|S_IROTH|S_IWOTH
            #define DEF_UMASK   S_IWGRP|S_IWOTH

            - DEF_MODE is the requested permissions, which were 

                Owner: read + write
                Group: read + write
                Others: read + write

            - DEF_UMASK defines the permissions to REMOVE

                Group: write permission removed
                Others: write permission removed

            - each process has a umask that is set by calling the umask() function:

                umask(DEF_UMASK);
            
            - we use mode & ~umask to set the final persmission bits (umask has 1s where permissions should be blocked, so ~umask flips those 1s to 0s)

                #define DEF_MODE    S_IRUSR|S_IWUSR|S_IRGRP|S_IWGRP|S_IROTH|S_IWOTH
                #define DEF_UMASK   S_IWGRP|S_IWOTH

                umask(DEF_UMASK);
                fd = Open("foo.txt", O_CREAT|O_TRUNC|O_WRONLY, DEF_MODE);

                -- now only the user has read and write permissions, even though originally everyone had them

        • A process closes an open file by calling the close() function with the fd as the argument.

    D. Reading and Writing Files

        • Applications perform input and output by calling the read() and write() functions

            * see readwrite.c to see a demo of read() and write() in application

        • In some situations, read and write transfer fewer bytes than the application requests 
        
            - such short counts do not indicate an error, but rather occur for a number of reasons

            (1) short read from a file: happens when you ask for more bytes than are left in the file, and returns the number of bytes actually read              
                EOF is indicated by returning 0 next time

            (2) short read from a terminal: happens because terminal input is line-buffered, returns the number of bytes typed before pressing Enter
